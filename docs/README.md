# Documentation

Reference for the template's API surface. This is a server-driven Inertia
application: the "API" is mostly HTTP routes that return Inertia pages or redirects,
plus only 4 real JSON endpoints. The docs below pin down contracts that would
otherwise have to be inferred from the sources.

## `api/`

| File | What's inside |
|---|---|
| [`api/json-endpoints.md`](api/json-endpoints.md) | The 4 real JSON/XHR endpoints: request, response shape, authentication (session + CSRF) |
| [`api/permissions-matrix.md`](api/permissions-matrix.md) | Route → permission map (two-level model: middleware + FormRequest) and error catalog (403/404/422/429 + domain checks) |
| [`api/response-conventions.md`](api/response-conventions.md) | Inertia shared props, paginator envelope, `/up` health, notes on versioning/naming |
| [`api/routes.snapshot.txt`](api/routes.snapshot.txt) | Snapshot of `php artisan route:list` (generated by `composer doc:routes`). Captured with the bot **enabled** (`BOT_ACTIVE=true`) — it's a superset: with the default `BOT_ACTIVE=false`, the 3 `bot-messages.*` routes are not registered |

## Bot

The Python ↔ Laravel integration contract (shared `messages.json` + the `bot_messages`
table, the inline-HTML rule, env variables) is in [`../modules/max-bot/README.md`](../modules/max-bot/README.md).
The registry JSON schema is [`../modules/max-bot/messages.schema.json`](../modules/max-bot/messages.schema.json).

## Keeping it current

- Route snapshot: `composer doc:routes` after changing `routes/web.php` (for a
  stable superset, capture it with `BOT_ACTIVE=true` so the bot routes don't "jump"
  in the diff depending on the flag).
- The bot registry schema is verified by the test `modules/max-bot/tests/test_registry_schema.py`.
- The other docs are edited by hand — they're small and live next to the code they describe.

> Heavy tooling (Scribe, a generated OpenAPI portal, Postman collections) is
> deliberately **not** wired in: for a template this size, 4 JSON endpoints are
> cheaper to describe by hand. If the project grows an external public API, set it
> up under a separate `/api/v1` (see `api/response-conventions.md`) and only then
> bring in a generator.
